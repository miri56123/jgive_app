## ASSISTANT

Not necessary, and here's the reasoning:

**`amount` is correct as-is.** It means "the unit charge amount" — for one-time donations it's the full charge, for recurring it's the monthly installment. Both interpretations are consistent: it's always "what gets charged per transaction."

**`total_committed_amount` is a derived value** — it's `amount × months` for recurring, `amount` for one-time. Storing a column you can compute in one line violates normalization and creates a consistency risk: what if you update `months` but forget to update `total_committed_amount`? Then your data lies.

The clean solution is a model method, not a column:

```ruby
def total_committed_amount
  recurring? ? amount * months.to_i : amount
end
```

**Renaming `amount` → `monthly_amount` would be wrong** — it breaks the semantics for one-time donations. A one-time donation doesn't have a "monthly amount."

---

**When you WOULD want a separate column:**
- If you integrate a real payment provider like Stripe — you'd store the Stripe subscription ID separately, and the recurring charge amount is already in the Stripe subscription object, not your DB
- If you need to run financial reports across millions of rows and can't afford the `amount * months` computation — but even then, a DB-computed column is better than manual duplication

**Bottom line:** Keep `amount` as is, add `total_committed_amount` as a model method if you need it anywhere in the UI. No migration needed.

## ASSISTANT

Two real-world scenarios where you'd want both columns stored:

**1. Payment provider charges differently per frequency**
When you integrate Stripe, a recurring donation creates a `Subscription` that charges `amount` every month. A one-time donation creates a single `PaymentIntent` for the full amount. If a donor later calls and asks "what did I commit to?", your support team needs to see both the monthly charge AND the total commitment without doing mental math. At scale, you'd store:

```ruby
monthly_amount:         180   # what Stripe charges each month
total_committed_amount: 6480  # what the donor "promised" over 36 months
```

**2. Reporting and analytics**
If your finance team runs queries like:
- "Total commitments raised this month" (includes future recurring charges)
- "Total actually collected" (only completed charges)

Then `SUM(amount × months)` across millions of rows is expensive and fragile. A stored `total_committed_amount` makes these queries a single `SUM(total_committed_amount)`. This is the classic "materialize the derived value for read performance" pattern.

---

**Why neither applies here:**

- No Stripe integration — `amount` is never actually charged, it's just recorded
- 2 campaigns, dozens of donations — no reporting scale concern
- The assignment explicitly scopes out payment processing

So for this app: model method is the right call. If a real payment provider gets wired up later, that's the moment to add the column — alongside the subscription ID, webhook state, etc. — as part of that feature, not now.

  ```ruby
  require "net/http"
  class ExchangeRateService
    API_URL   = "https://api.frankfurter.dev/v1/latest"
    CACHE_TTL = 1.hour
    def self.to_ils(currency)
      return 1.0 if currency == "ILS"
      Rails.cache.fetch("exchange_rate/#{currency}_to_ILS", expires_in: CACHE_TTL) do
        uri  = URI("#{API_URL}?from=#{currency}&to=ILS")
        http = Net::HTTP.new(uri.host, uri.port)
        http.use_ssl      = true
        http.open_timeout = 5
        http.read_timeout = 5
        rate = JSON.parse(http.get(uri.request_uri).body).dig("rates", "ILS").to_f
        rate.positive? ? rate : 1.0
      end
    rescue => e
      Rails.logger.warn("ExchangeRateService: could not fetch #{currency}→ILS rate: #{e.message}")
      1.0
    end
  end
  ```

  ```ruby
  def self.to_ils(currency, force: false)
    return 1.0 if currency == "ILS"
    Rails.cache.fetch("exchange_rate/#{currency}_to_ILS", expires_in: CACHE_TTL, force: force) do
      uri  = URI("#{API_URL}?from=#{currency}&to=ILS")
      http = Net::HTTP.new(uri.host, uri.port)
      http.use_ssl      = true
      http.open_timeout = 5
      http.read_timeout = 5
      rate = JSON.parse(http.get(uri.request_uri).body).dig("rates", "ILS").to_f
      rate.positive? ? rate : 1.0
    end
  rescue => e
    Rails.logger.warn("ExchangeRateService: could not fetch #{currency}→ILS rate: #{e.message}")
    1.0
  end
  ```

  ```ruby
  class CreateDonation
    Result = Struct.new(:success, :donation, :errors, keyword_init: true) do
      def success? = success
    end

    def initialize(campaign:, params:)
      @campaign = campaign
      @params   = params
    end

    def call
      donation = @campaign.donations.build(@params)
      donation.exchange_rate = ExchangeRateService.to_ils(donation.currency)

      if donation.save
        Rails.logger.info("[CreateDonation] created id=#{donation.id} campaign=#{@campaign.id} amount=#{donation.amount} #{donation.currency}")
        Result.new(success: true, donation: donation, errors: [])
      elsif (existing = duplicate(donation))
        Rails.logger.info("[CreateDonation] duplicate prevented key=#{donation.idempotency_key} existing_id=#{existing.id}")
        Result.new(success: true, donation: existing, errors: [])
      else
        Rails.logger.warn("[CreateDonation] validation failed campaign=#{@campaign.id} errors=#{donation.errors.full_messages}")
        Result.new(success: false, donation: donation, errors: donation.errors.full_messages)
      end
    end

    private

    def duplicate(donation)
      return nil if donation.idempotency_key.blank?
      return nil unless donation.errors[:idempotency_key].any?
      Donation.find_by(idempotency_key: donation.idempotency_key)
    end
  end
  ```
